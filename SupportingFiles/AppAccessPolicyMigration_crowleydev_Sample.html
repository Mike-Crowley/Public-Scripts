To constrain an unconstrained app: create a management scope for the mailboxes it actually needs, add the matching Application * RBAC role assignment (pattern in any Ready row above), verify with Test-ServicePrincipalAuthorization, then revoke the tenant-wide Entra grant. Order matters: RBAC first, revoke second — the app never loses access it should have.

Migration Steps (per app):
1. New-ServicePrincipal — Exchange pointer to the Entra service principal
2. New-ManagementScope — mailbox scope (MemberOfGroup DN filter, or ExternalDirectoryObjectId for a single mailbox)
3. New-ManagementRoleAssignment — assign RBAC application roles with the scope
4. Test-ServicePrincipalAuthorization — verify InScope = True, and test the app itself
5. Revoke only the migrated permission grants in Entra ID (per-permission via Graph; the portal's "Revoke admin consent" button revokes everything at once). Removing rows from the app registration's API-permissions list afterwards is hygiene only — it does not revoke access. Permissions marked keep are not replaced by RBAC and must stay.
6. Remove-ApplicationAccessPolicy — delete the legacy policy last

Caveats: RBAC scoping only takes effect once the tenant-wide Entra grant is revoked (grants are a union). Exchange caches app permissions for 30 minutes–2 hours; allow for that between steps 5 and 6. MemberOfGroup scopes match direct group members only — flatten nested groups first. DenyAccess policies have no RBAC equivalent and must not be migrated with these steps. Policies and RBAC apply to app-only (application) permissions — apps with only delegated Exchange permissions never needed a policy. IMAP/POP app permissions have no RBAC roles; that access is controlled via Add-MailboxPermission. EWS (Application EWS.AccessAsApp) is blocked for non-Microsoft apps starting Oct 1, 2026 and removed after Apr 2027 — plan Graph migrations for EWS apps.
